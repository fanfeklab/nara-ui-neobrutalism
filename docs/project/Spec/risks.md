# Technical Risks & Mitigation

| Risk | Likelihood | Impact | Mitigation |
|:---|:---|:---|:---|
| **Kompleksitas Refine & Customization**<br>Refine memiliki learning curve yang cukup curam, terutama saat ada kustomisasi UI di luar flow standar. | Sedang | Tinggi | Gunakan Headless mode dari Refine dan padukan dengan komponen Shadcn UI yang kita miliki agar UI dapat 100% dikendalikan, alih-alih bergantung pada komponen UI bawaan Refine. |
| **Performa Canvas Seat Mapping (Konva.js)**<br>Event dengan kapasitas sangat besar (misal 50.000 seat) bisa menyebabkan lag pada browser jika di-render sekaligus dalam satu canvas. | Sedang | Sedang | Implementasi "Viewport Frustum Culling" atau *lazy rendering* di Konva.js (hanya render objek yang terlihat pada zoom level tertentu) atau kelompokkan seat ke dalam block/zona agar tidak semua seat dimuat secara detail secara bersamaan. |
| **Inkonsistensi State pada Tiered Approval (Finance)**<br>Proses transaksi persetujuan bersamaan oleh lebih dari 1 approver bisa menimbulkan state conflict. | Rendah | Tinggi | Terapkan *Optimistic Concurrency Control* di database, misalnya dengan field `version`, dan bungkus operasi approval dalam mekanisme *Database Transactions* yang strict. |
| **Keamanan Endpoint PWA offline-first**<br>Sinkronisasi data Check-in atau input field staff dari PWA saat offline bisa menjadi vector serangan manipulasi timestamp. | Sedang | Sedang | Implementasikan validasi *server-side timestamp* yang strict. Simpan `action_id` dan `signature` dari PWA untuk menjamin integritas urutan kejadian saat disinkronisasikan ke backend. |
| **API Versioning Mobile App Future Proofing**<br>Perubahan API web bisa merusak integrasi Mobile Apps native yang belum di-update oleh user. | Tinggi | Sedang | Definisikan API route dengan versioning dari awal (contoh: `/api/v1/event`). Ekstrak *business logic* ke service class independen agar bisa digunakan kembali oleh `/api/v2/...` tanpa mengganggu endpoint lama. |
